# cusa вҖ” Cursor-SDK-powered coding CLI with transparent auto-mode

## Overview

**Purpose.** Provide a terminal coding agent that leverages the Cursor SDK's multi-provider model access while making model selection **explicit and controllable per task** вҖ” replacing Cursor's opaque built-in auto mode with a transparent, deterministic router.

**Scope (included).**
- Interactive TUI coding agent (chat REPL) whose look-and-feel is derived from OpenAI's Rust `codex-tui` (Ratatui-based, Apache-2.0), forked and re-engined against the Cursor SDK.
- Custom **auto mode**: an in-process router that classifies each user turn and picks a Cursor SDK model (using `agent.send({ model })` per-turn override), displaying the chosen model in the UI before every turn.
- MCP server configuration and display, Cursor-format skills loader, Codex-style approval modes, `Agent.resume` session recovery, and cumulative token/cost telemetry.
- Distribution as a single `npm i -g` package that ships a platform-specific native TUI binary plus a Node.js sidecar hosting `@cursor/sdk`.

**Scope (excluded).**
- Cloud runtime as a v1 primary target (see Risks вҖ” local-first is v1; cloud comes later).
- IDE plugins, VS Code / Cursor extensions, or web frontends.
- Reimplementing Cursor SDK features (streaming, MCP, subagents) вҖ” we consume them via `@cursor/sdk`, we do not fork the SDK.
- Multi-user / team workspace UI. (Single-user local CLI only.)
- Codex's cloud API integration or ChatGPT account flows.

**Working name.** `cusa` (throwaway, may be renamed before v1).

## Glossary

| Term | Definition |
|------|------------|
| **TUI** | Terminal User Interface. In this project, the Rust/Ratatui process forked from `codex-tui`. |
| **Sidecar** | The Node.js process spawned by the TUI that hosts `@cursor/sdk` and executes agent runs. |
| **Bridge** | The JSON-RPC-over-stdio channel between the TUI and the sidecar. |
| **Turn** | One `agent.send()` call and its resulting `Run`. |
| **Auto mode (this project)** | Deterministic in-process router that picks a Cursor SDK model per turn. Distinct from Cursor's built-in server-side `auto`. |
| **Router** | The subsystem that classifies a user turn and returns a `ModelSelection`. May be rule-based, LLM-based, or hybrid. |
| **Approval mode** | Codex-style gating of tool calls: `suggest` (prompt for every action), `auto-edit` (auto-approve file reads, prompt for writes/shell), `full-auto` (auto-approve everything, sandboxed). |
| **Skill** | A Markdown document at `~/.cursor/skills/**/SKILL.md` following Cursor's skill format. Injected into the agent's context on demand. |
| **Session** | A durable Cursor SDK agent (identified by `agentId`) plus its cusa-local state (skills enabled, approval mode, router config). |

## User Stories

- **US-001.** As a solo developer, I can start `cusa` in a repo and chat with an agent that streams responses to my terminal.
- **US-002.** As a solo developer, I can see the model name displayed before every turn so I know what generated the answer.
- **US-003.** As a power user, I can edit `~/.cusa/router.toml` to change how prompts are classified and mapped to models.
- **US-004.** As a power user, I can override the router for a single turn with `/model <id>`.
- **US-005.** As a Cursor-ecosystem user, I can toggle any discovered skill on/off for the current session and have enabled skills injected into agent context.
- **US-006.** As a Cursor-ecosystem user, I can see which MCP servers are active and which tools they expose.
- **US-007.** As a security-conscious user, I can choose an approval mode and have file writes / shell commands gated accordingly.
- **US-008.** As a returning user, I can resume yesterday's session by picking it from a list at startup or via `--resume <agentId>`.
- **US-009.** As a cost-conscious user, I can see cumulative and per-turn token counts in the status line and via `/cost`.
- **US-010.** As a first-time user, I can install with `npm i -g cusa` and run `cusa` вҖ” no Rust toolchain, no Python, just Node вүҘ 20.

## Functional Requirements

### FR-1. Interactive chat loop
- Streaming TUI matching Codex-TUI visual conventions (status line, transcript pane, input pane, footer hints).
- Slash commands: `/help`, `/clear`, `/reset`, `/model`, `/mode`, `/approval`, `/skills`, `/mcp`, `/cost`, `/resume`, `/quit`.
- Multi-line input toggle (Shift+Enter or `\`-continuation).
- Command history navigation (Up/Down).
- Ctrl-C cancels the active run without exiting the app; Ctrl-C twice exits.

### FR-2. Custom auto mode (router)
- On every non-command user input, the router receives `{ prompt, conversationSummary, currentModel, sessionMetadata }` and returns `{ model: ModelSelection, rationale: string }`.
- The chosen model and 1-line rationale are rendered in the TUI **before** the run starts.
- Router pipeline (v1): (a) fast rule pass (regex/keywords/prompt length/mode) вҶ’ if confident, return; (b) otherwise fall through to a small LLM classifier call (default: `composer-2.5` with a short classification prompt) вҶ’ return.
- Router config file `~/.cusa/router.toml` with editable rule list and default fallback model.
- If the router itself fails (timeout, error), fall back to the session default model and surface a warning in the UI.
- User overrides: `/model <id>` sets a sticky override until the user runs `/model auto`.

### FR-3. Approval modes
- Three modes: `suggest`, `auto-edit`, `full-auto` вҖ” semantics inherited from Codex CLI's approval concept but implemented over Cursor SDK's stream events.
- Mode selector: `/approval` command and a footer shortcut key.
- In `suggest`, every tool call surfaces a Y/N prompt with the tool name and arguments; user can approve/deny/always-approve-this-tool for the session.
- In `auto-edit`, read-only tools are auto-approved; write and shell-exec tools prompt.
- In `full-auto`, all tools auto-approved. Local sandbox (SDK `local.sandboxOptions.enabled = true`) is enabled automatically when this mode is selected.

### FR-4. Skills loader (Cursor format)
- Discover skills recursively from `~/.cursor/skills/**/SKILL.md` and `<repo>/.cursor/skills/**/SKILL.md`.
- Parse each skill's frontmatter (`name`, `description`).
- `/skills` opens a togglable list; changes take effect on the next turn.
- Enabled skills are injected as `<skill name="вҖҰ"><description>вҖҰ</description><body>вҖҰ</body></skill>` blocks into the system context passed on each `agent.send()`.
- Skills disabled by default at session start (avoid context bloat); user or session config can pre-enable a list.

### FR-5. MCP configuration
- Layered load (matching SDK precedence): inline (`--mcp` CLI flag or `~/.cusa/mcp.overrides.json`) вҶ’ project `.cursor/mcp.json` вҶ’ user `~/.cursor/mcp.json`.
- The `local.settingSources` passed to the SDK is `["user", "project"]` by default; overridable via `--setting-sources`.
- `/mcp` displays each active server with its status, transport, and enumerated tools (fetched via SDK tool listing).
- MCP servers can be toggled per session; toggling triggers a run-end + `agent.send()` re-arm with the new server set on the next turn (inline `mcpServers` fully replace creation-time servers per SDK docs).

### FR-6. Session persistence & resume
- Every agent created records `{ agentId, cwd, model, approvalMode, enabledSkills, mcpOverrides, createdAt, lastUsedAt }` to `~/.cusa/sessions.json`.
- Startup: begins a new session by default. With `--resume-picker`, if `sessions.json` has entries for the current `cwd`, show a "Resume last / Pick from list / New session" chooser.
- `--resume <agentIdOrIndex>` bypasses the chooser.
- Resume calls `Agent.resume(agentId)` and re-attaches inline MCP servers from the stored overrides.

### FR-7. Cost / telemetry
- Persistent status-line region shows cumulative `TokenUsage` (in/out/cache/total) for the current session.
- After every turn, a one-line summary shows delta usage for that turn.
- `/cost` command opens a detail pane with per-turn history and per-model aggregates.
- Values sourced from the SDK's `run.usage` / `result.usage` and the `SDKUsageMessage` stream event.

### FR-8. Local-context multi-turn workaround
- Because local `agent.send()` currently does not retain conversation context (confirmed Cursor bug вҖ” see Risks R-1), each `send()` prepends a compact conversation summary.
- Two summary strategies: (a) last-N-turns raw replay with a configurable N (default 6), (b) LLM-summarized rolling context when raw replay exceeds a byte budget (default 32 KiB).
- Strategy switches automatically based on the byte budget; user can force one via `/context strategy=raw|summary`.
- When Cursor ships the native fix, the workaround becomes opt-in via config.

### FR-9. Architecture: Rust TUI + Node.js sidecar
- The npm package `cusa` installs a platform-detected native TUI binary (Rust) plus a bundled Node.js sidecar script.
- On `cusa` launch, the TUI spawns the sidecar (`node <install>/sidecar/index.js`), performs a JSON-RPC handshake over stdio, and holds it for the lifetime of the session.
- All Cursor SDK calls (Agent.create, send, resume, cancel, list models) are made by the sidecar; the TUI only issues typed JSON-RPC requests and consumes typed stream events.
- Sidecar owns `CURSOR_API_KEY` reading and never returns the key over the RPC channel.
- TUI supervises the sidecar (health ping every 5 s; auto-restart on unexpected exit; surface fatal errors to the user).

### FR-10. Distribution
- Single npm package: `npm install -g cusa`.
- `postinstall` script downloads the correct native TUI binary from GitHub Releases based on `process.platform` + `process.arch`.
- Bundled `sidecar/` directory ships as plain JS + `package.json` with pinned `@cursor/sdk` version.
- No Rust toolchain, no Python, no Homebrew requirement for end users. Node.js вүҘ 20 required.
- Attribution: `THIRD_PARTY_NOTICES.md` includes upstream Codex (Apache-2.0) notice for any file forked from `codex-rs/tui`.

## Main Flow

### 5.1 First-time launch in a repo
1. User runs `cusa` in `~/projects/foo`.
2. TUI starts, spawns sidecar, sidecar reports Ready with SDK version, resolves model list via `Cursor.models.list()`.
3. TUI discovers skills (found: 12), MCP servers (found: 3), asks: "No prior sessions here. Start new? [Y/n]".
4. User accepts. TUI displays welcome screen with default model (from config), approval mode `suggest`, no skills enabled.
5. User types a prompt. Router classifies (rule-based hit: "explain code" вҶ’ `composer-2.5`). Line renders: `вҶ’ composer-2.5 В· fast rule: explain code`.
6. Sidecar calls `Agent.create` with the picked model and `settingSources: ["user", "project"]`, then `agent.send(prompt + context)`.
7. Stream events flow TUI вҶҗ sidecar; text renders live; tool calls appear as gated approval dialogs per approval mode.
8. Turn ends. TUI shows delta tokens + model used; status line updates cumulative.

### 5.2 Resume flow
1. `cusa --resume-picker` in the same repo; startup finds 2 prior sessions.
2. TUI shows chooser: "1. yesterday 18:04 В· claude-sonnet-4 В· 42 turns В· 210 K tokens / 2. this morning 09:12 В· composer-2.5 В· 4 turns / N. New session".
3. User picks 1; TUI calls `Agent.resume(agentId)` via sidecar; enabled skills, approval mode, MCP overrides restored from `sessions.json`.
4. `agent.model` is `undefined` after resume (per SDK); TUI prompts router on the next turn to re-establish selection or accepts stored `sessionModel`.

### 5.3 Auto-mode override flow
1. User types `/model claude-sonnet-4`. Sticky override recorded.
2. Router bypassed for subsequent turns; UI shows `вҶ’ claude-sonnet-4 В· manual override`.
3. `/model auto` clears the override.

## Non-Functional Requirements

### NFR-1. Performance
- Cold start (npm-installed) to first prompt: **вүӨ 800 ms** on M2 MacBook, warm shell.
- Sidecar spawn + handshake: **вүӨ 400 ms**.
- Router classification (rule path): **вүӨ 5 ms**.
- Router classification (LLM path, fallback): **вүӨ 1500 ms p95** using `composer-2.5`.
- Streaming: first assistant token to TUI within **вүӨ 100 ms** of sidecar receiving it (best-effort вҖ” dominated by SDK).

### NFR-2. Security
- `CURSOR_API_KEY` read from env or from `~/.cusa/config.toml` `api_key = "вҖҰ"`; never printed to TUI, never sent in RPC responses.
- MCP inline overrides may contain secrets; stored on disk only if the user explicitly runs `/mcp save`, and file mode enforced 0600.
- No telemetry to third parties. `~/.cusa/sessions.json` is local-only.
- `full-auto` approval mode requires `local.sandboxOptions.enabled = true`.

### NFR-3. Monitoring / observability (user-facing)
- Cumulative and per-turn token usage always visible.
- `--verbose` prints RPC frames and stream events to a rotating log at `~/.cusa/logs/`.
- Fatal sidecar errors surface as a red modal with a copyable log path.

## Edge Cases

- **No API key.** TUI shows a first-run onboarding: "Set `CURSOR_API_KEY` or run `cusa login`". `cusa login` writes the key to `~/.cusa/config.toml` after confirmation.
- **Sidecar crashes mid-run.** TUI marks the run as failed, offers `Retry / New turn`, restarts sidecar in background.
- **Network dies mid-stream.** SDK throws or returns `result.status == "error"`; TUI shows the error, keeps transcript, allows retry.
- **Router LLM fails.** Fall back to session default model, surface warning line, do not block the user.
- **User picks a model they don't have access to.** SDK returns `CursorAgentError` before run starts; TUI surfaces the actual message, lists available models via cached `models.list`.
- **MCP server takes > 10 s to start.** TUI shows a spinner in `/mcp`; on timeout, mark the server as failed and allow toggle-retry.
- **Two `Ctrl-C` in вүӨ 500 ms.** Force-quit: cancel run (if any), dispose agent, kill sidecar, exit.
- **Skills file has invalid frontmatter.** Skip that file, log a warning; do not crash discovery.
- **Cursor SDK fixes local context bug mid-life.** Feature-detect via SDK version; disable manual history injection.
- **Long-running turn (> 5 min).** No client timeout; user can Ctrl-C. Sidecar honors `err.retry_after` before any automatic retry.

## Concurrency & Cancellation

- One active `agent.send()` at a time per session. Second submit while streaming is queued or rejected (config: `queue_mode = queue | reject`; default `reject` with a message).
- Ctrl-C on active run: TUI issues JSON-RPC `cancel`; sidecar calls `run.cancel()` if `run.supports("cancel")`; TUI waits for `run.wait()` to settle (either finished-with-partial or cancelled). If wait exceeds 3 s, TUI marks the run as detached and moves on; sidecar continues to reap the underlying handle.
- MCP server list is immutable during a run вҖ” toggles queue until the next `send()`.

## Network / Timeout

- No hard client-side timeout on run duration. Sidecar keeps the run alive until `wait()` resolves or user cancels.
- Router LLM call has a 5 s hard timeout; on timeout, fall back to rules-only path.
- MCP HTTP transports honor SDK-configured retries; cusa does not override.

## Data Flow

**Send-turn happy path:**
1. User submits prompt in TUI вҶ’ TUI RPC `session/send { text }`.
2. Sidecar `router.route(prompt, state)` вҶ’ `ModelSelection`.
3. Sidecar emits RPC notification `router/decision { model, rationale }`.
4. TUI renders the router line.
5. Sidecar builds context: enabled skills + history workaround summary + prompt.
6. Sidecar `agent.send(text, { model, mcpServers, onStep, onDelta })`.
7. Stream events: sidecar вҶ’ TUI as `stream/message` and `stream/usage` notifications.
8. Tool calls: sidecar emits `tool/approvalRequest`; TUI resolves and replies with `tool/approvalResponse`; sidecar applies decision to the run.
9. `run.wait()` returns; sidecar emits `run/finished { usage, result }`; TUI updates cost region.

## ADR / Trade-offs

### ADR-01. Rust TUI + Node.js sidecar over a monolithic Node CLI
- **Chosen**: Rust TUI (from `codex-tui`) + Node sidecar.
- **Alternative rejected**: Full-Node ink CLI. Rejected because user explicitly wants to fork the current Rust TUI to preserve modern Codex UX and rendering performance.
- **Trade-offs**: Adds cross-language complexity, IPC overhead (bounded, вүӨ few ms per frame), and postinstall binary distribution. Buys: exact Codex look/feel, Ratatui rendering, best-in-class terminal input handling, and future upstream cherry-picks.

### ADR-02. Router = rule pass + LLM fallback (hybrid)
- **Chosen**: Fast deterministic rules first, LLM classifier only when rules yield low confidence.
- **Alternative rejected**: Pure LLM router. Rejected due to per-turn cost and 1вҖ“2 s latency added to every turn.
- **Alternative rejected**: Pure rule router. Rejected due to inability to handle open-ended prompts well.
- **Trade-offs**: Hybrid adds config complexity but keeps common-case latency near zero.

### ADR-03. Local-context workaround via history replay
- **Chosen**: Prepend last-N-turns replay (or LLM summary when over budget) to each `agent.send()`.
- **Alternative rejected**: Switch to cloud runtime. Rejected because user chose local-first UX.
- **Trade-offs**: Prompt bloat, tool-call context loss (best-effort вҖ” tool outputs replayed as text). To be removed when Cursor ships native local context retention.

### ADR-04. Own JSON-RPC schema (not Codex's app-server-protocol)
- **Chosen**: Small purpose-built JSON-RPC schema owned by cusa.
- **Alternative considered**: Reuse `codex-app-server-protocol` and implement it in Node.
- **Trade-offs**: Own schema keeps us free to evolve independently. Cost: cannot piggyback on Codex's future IDE-integration protocol changes. Revisit at v2.

## Dependencies & Assumptions

**Runtime dependencies.**
- `@cursor/sdk` (TypeScript, current beta). Version pinned in `sidecar/package.json`.
- Node.js вүҘ 20 on the user's machine.
- OS: macOS (arm64, x64), Linux (x64, arm64), Windows (x64). No 32-bit.
- Terminal with truecolor and Unicode support (matches Codex-TUI baseline).

**Upstream dependencies.**
- OpenAI `codex-rs/tui` вҖ” Apache-2.0. We fork the TUI crate (and any minimally required utility crates) into `cusa-rs/tui`. All forked files retain their license headers; `THIRD_PARTY_NOTICES.md` documents attribution.

**Assumptions.**
- **A1.** Cursor SDK's per-turn model override (`agent.send({ model })`) remains stable in the beta. Verified from official TypeScript SDK docs.
- **A2.** Token usage from `run.usage` / `result.usage` / stream `usage` event remains available. Verified from official SDK docs.
- **A3.** `local.settingSources` gates MCP file layering but does **not** currently auto-inject skills. cusa loads skills itself by walking the filesystem and injecting into the system context. (Assumption user-waived: user chose Cursor-format skills only; if SDK later gains native skill loading, we deprecate ours.)
- **A4.** `Agent.resume` re-uses stored conversation server-side for cloud but re-hydrates a local agent's handle for local runs. Inline MCP servers must be re-passed on resume.
- **A5.** JSON-RPC over stdio is stable enough for our needs (Codex `app-server` uses the same pattern). No SSE / websocket needed.
- **A6.** The Codex-TUI subset we need can be forked cleanly without pulling all of `codex-core`. If not, plan pivots to "replicate look/feel with Ratatui" (see Risks R-3).

## Risks & Open Questions

| ID | Risk | Mitigation |
|----|------|------------|
| R-1 | Cursor local-agent multi-turn context bug. Confirmed by Cursor staff on forum; no ETA. | FR-8 workaround: manual history injection. Feature-flag to disable once fixed. |
| R-2 | `@cursor/sdk` is beta; breaking changes likely. | Pin exact version; add release-notes watcher in CI; expose `--sdk-version` and log on startup. |
| R-3 | Forking `codex-tui` may pull in more codex-core coupling than expected. | Time-box the fork at 2 weeks; fall back to a from-scratch Ratatui UI that visually matches Codex. Document as ADR-01a if triggered. |
| R-4 | Router LLM classifier adds per-turn cost the user didn't ask for. | Hybrid design keeps LLM path off the hot path for most turns. Expose `router.llm_enabled = false` config flag. |
| R-5 | Skills injection may exceed model context. | Cap total injected skill text (default 16 KiB), warn user on truncation. |
| R-6 | npm postinstall download unreliable in some corporate networks. | Cache binary under `~/.cusa/bin/`; provide `cusa download-binary` command; document proxy env vars. |
| R-7 | `full-auto` + sandbox on Windows: SDK sandbox support is Linux/macOS. | Detect platform; disable `full-auto` on Windows with a message. |
| R-8 | License / attribution mistakes when forking Codex files. | `THIRD_PARTY_NOTICES.md` + a `scripts/check-headers.sh` CI script that fails if forked files lose their license header. |

## Acceptance Criteria

- **AC-1.** Given a repo with no prior sessions, when I run `cusa` with a valid `CURSOR_API_KEY`, then within 800 ms I see a Codex-styled TUI with cursor ready for input.
- **AC-2.** Given a prompt "explain what this function does", when I press Enter, then the TUI displays a router decision line **before** streaming starts, naming the chosen model.
- **AC-3.** Given I type `/model claude-sonnet-4`, when I submit any subsequent prompt, then the router is bypassed and the router line reads `manual override`.
- **AC-4.** Given approval mode `suggest`, when the agent attempts to write a file, then a Y/N prompt appears with the tool name and file path before the write executes.
- **AC-5.** Given a `.cursor/mcp.json` in the project, when I open `/mcp`, then all its servers appear with their transports and tools enumerated.
- **AC-6.** Given `~/.cursor/skills/foo/SKILL.md` exists, when I open `/skills` and toggle `foo` on, then the next turn's `agent.send()` includes the skill body in the system context.
- **AC-7.** Given I complete two turns totaling 5000 tokens, when I run `/cost`, then the pane shows per-turn breakdown summing to 5000.
- **AC-8.** Given I quit and re-run `cusa`, when the chooser appears, then I can pick the previous session and continue with restored approval mode / skills / MCP overrides.
- **AC-9.** Given I press Ctrl-C mid-stream, when I confirm, then the run is cancelled within 3 s and control returns to the input pane.
- **AC-10.** Given the sidecar crashes, when the TUI detects the exit, then within 2 s a modal appears offering `Retry / Quit` and a copyable log path.
- **AC-11.** Given the router LLM classifier times out (> 5 s), when the turn continues, then the session default model is used and a warning is logged inline.
- **AC-12.** Given I install with `npm i -g cusa` on a supported platform, when I run `cusa --version`, then it prints a version and does not require any additional install step.
